## 0. 通用约束（必须遵守，避免破坏现有系统）

**技术栈**：前端 React 18 + Vite；后端 Express 4 + TypeScript（tsx）；Prisma 5 + PostgreSQL 16；Redis 7 + BullMQ 5；sharp 图片处理；fabric v5.5.2（CJS，`import fabricDefault from 'fabric'`，fallback `window.fabric`，运行时 `new fabric.Canvas`）。

**既有铁律（来自 MEMORY.md / 开发需求文档 §7）**：
1. **积分**：任何消耗积分的功能，必须走 `credit.service` 的统一冻结→消费→退款→流水链路，禁止绕过。新增积分类型在 `SystemSetting` 的 credit-rules 中可配置。
2. **所有权校验**：涉及 user/project/asset/task/export 的接口必须做归属校验，防 IDOR（参考已有 P0 修复）。
3. **RBAC 判定**：管理员 = `user.role==='admin'` **或** `user.roles[].role.code==='admin'`（数组判定，勿用单数 `user.role` 硬判）。
4. **Worker 独立进程**：生成/合成/重绘/核价等耗时任务必须进 BullMQ 队列，由 `pnpm --filter server worker`（生产 `worker:prod`）消费；原生 `pnpm dev` 不起 Worker。
5. **迁移规则**：新增表/列用 `prisma migrate diff --from-migrations ./prisma/migrations --to-schema-datamodel ./prisma/schema.prisma --shadow-database-url=... --script` 生成 SQL，手写迁移目录；生产用 `migrate deploy`，**禁止 `db push`**（会丢 `@map` 触发漂移）。`SystemSetting.updatedAt` 必须 `@default(now()) @map("updated_at")`。
6. **环境守卫**：`config/index.ts` 生产要求 `JWT_SECRET` 非空非默认、`CORS_ORIGINS` 非 localhost；改动后端后跑 `cd apps/server && ./node_modules/.bin/tsc --noEmit`。
7. **测试**：开发/改动前读取 `.workbuddy/memory/TESTING-PLAYBOOK.md`，按 §2 补对应测试（单测/接口/集成/冒烟/E2E/安全/性能）。断言以 DB/积分真值为准。

**新增模块统一落点**：后端 `apps/server/src/{routes,services,workers}/<module>/`；前端 `apps/web/src/{pages,components,api}/<module>/`；数据模型在 `apps/server/prisma/schema.prisma` 增量追加。

## 1. M2 权限与组织（字段级权限落地 + 操作审计日志）【P0】

### 1.1 目标
把「老板独有字段（预算/定金/余额/客户电话/微信/地址）」做成**字段级权限**强制生效，并补齐**管理员操作审计日志**。

### 1.2 功能点
- 字段级权限规则表可配置：字段 → 可见角色集合（如 `balance,deposit,budget,clientPhone,clientWechat,clientAddress` 仅 `boss/admin` 可见）。
- 后端在序列化 User/Project/Order 等含敏感字段的响应时，按 `req.user` 角色剥离不可见字段（白名单反转）。
- 前端按权限渲染字段（老板可见，设计师/工人不可见，且不暴露字段名）。
- 管理员操作（加减积分、改角色、改模板、调配置）写入 `AdminAuditLog`。

### 1.3 数据模型（增量）
```prisma
model FieldPermission {
  id        Int      @id @default(autoincrement())
  entity    String   // 如 "User" | "Project" | "Order"
  field     String   // 如 "balance"
  roles     String   // JSON 数组，如 ["boss","admin"] 表示仅这些角色可见
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  @@unique([entity, field])
}

model AdminAuditLog {
  id        Int      @id @default(autoincrement())
  operatorId Int
  operator  User     @relation(fields: [operatorId], references: [id])
  action    String   // 如 "credit.adjust" | "role.update" | "template.delete"
  targetType String?
  targetId  Int?
  detail    Json?    // 变更前后
  ip        String?
  createdAt DateTime @default(now())
  @@index([operatorId, createdAt])
}
```

### 1.4 接口契约
- `GET /api/admin/field-permissions` → 列表（仅 admin）
- `PUT /api/admin/field-permissions` body `{entity,field,roles[]}` → upsert
- 现有 `GET /api/projects/:id`、`GET /api/admin/users` 响应中敏感字段按权限剥离（**向后兼容：缺失规则时默认全部可见，避免回归**）
- `GET /api/admin/audit-logs?page=&action=` → 审计列表（仅 admin）

### 1.5 关键流程
请求带敏感字段的接口 → `fieldPermission.middleware` 取 `req.user.roles` → 查 FieldPermission → 用白名单反转 `omit` 不可见字段 → 返回。

### 1.6 验收标准
- 设计师/工人调用 `GET /api/admin/users` 或项目详情，响应中**不含** `balance/deposit/budget/clientPhone/clientWechat/clientAddress` 字段。
- 老板/admin 可见全部字段。
- 管理员加减积分后，`AdminAuditLog` 新增一条 `action=credit.adjust` 记录，含前后值。

### 1.7 测试场景
- **安全（IDOR/越权）**：以设计师 token 请求含敏感字段接口，断言响应无敏感字段（必含反例）。
- **单元**：`omitSensitiveFields(data, roles)` 纯函数测试（含规则缺失默认可见、多规则叠加）。
- **集成**：admin 改角色→断言审计日志写入；设计师越权改他人积分→断言 403。
- **回归**：老板视角字段完整，前端不报错。

### 1.8 上线检查
- 迁移 `migrate deploy` 后，`FieldPermission` 预置种子（老板独有字段规则）。
- 前端权限组件 `<CanField role="boss">` 全局替换硬编码敏感字段渲染。

## 2. M4 品牌资产库（Brand Kit / DAM）【P0】

### 2.1 目标
为客户/企业建立品牌套件（色板/字体/Logo/语调），并沉淀可商用素材与标签，AI 生图/套版默认套用 VI。

### 2.2 功能点
- 品牌套件 CRUD：名称、主色/辅色（hex 数组）、字体（标题/正文）、Logo 资产、品牌语调（brand voice 文本）。
- 客户 VI 绑定到 Project（项目创建时选品牌）。
- 素材库：上传/生成素材归入品牌，打标签（自动+手动），支持检索。
- 生图 prompt 自动注入品牌语调与配色（在 AI brief/prompt 阶段拼接）。
- 版权/授权字段（source、license、author），溯源展示。

### 2.3 数据模型
```prisma
model BrandKit {
  id          Int      @id @default(autoincrement())
  ownerId     Int
  owner       User     @relation(fields: [ownerId], references: [id])
  name        String
  primaryColors Json   // ["#FFF","#000"]
  fontTitle   String?
  fontBody    String?
  logoAssetId Int?
  brandVoice  String?  // 语调描述，注入 prompt
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  projects    Project[]
  @@index([ownerId])
}

model AssetTag {
  id       Int    @id @default(autoincrement())
  assetId  Int
  asset    Asset  @relation(fields: [assetId], references: [id], onDelete: Cascade)
  tag      String
  source   String @default("manual") // manual | auto_ocr | auto_vision
  @@unique([assetId, tag])
  @@index([tag])
}
// Asset 表追加字段：
// license   String?  // 如 "CC-BY" | "commercial" | "owned"
// author    String?
// brandKitId Int?
```

### 2.4 接口契约
- `POST /api/brand-kits` / `GET /api/brand-kits` / `PUT /api/brand-kits/:id` / `DELETE /api/brand-kits/:id`
- `POST /api/projects` / `PATCH /api/projects/:id` body 增加 `brandKitId`
- `POST /api/assets/:id/tags` / `GET /api/assets?tag=&brandKitId=`
- `GET /api/brand-kits/:id/prompt-injection` → 返回拼好的品牌上下文片段（供 AI 服务调用）

### 2.5 验收标准
- 创建品牌套件后，新建项目绑定该品牌，调用 AI brief 时 prompt 自动含品牌语调+配色。
- 素材可打标检索；版权字段在详情展示。

### 2.6 测试场景
- **单元**：`buildBrandPromptInjection(brandKit)` 输出拼接正确。
- **集成**：绑定品牌→AI brief 请求体含品牌字段（mock AI 网关断言）。
- **接口**：标签 CRUD + 检索分页；归属校验（不能打他人资产标签）。
- **E2E**：建品牌→建项目→生图→效果含品牌色（截图比对或 prompt 断言）。

### 2.7 上线检查
- 迁移含 `AssetTag` 唯一约束与索引；`brand_kits` 索引。
- 前端「品牌」入口（老板/管理员可见），与项目创建表单联动。

## 3. M5 模板市集 + 源文件转模板【P0】

### 3.1 目标
让设计师把交付稿「一键转模板」沉淀到模板库，支持分类检索、分享、套用；后台可运营模板市集。

### 3.2 功能点
- 「保存为模板」：从 ProjectVersion.canvasJson 生成 Template（含可编辑占位符标记）。
- 模板分类（businessType / category）、封面、搜索、排序。
- 模板「套用」：新建项目并载入模板 canvasJson（占位符可二次编辑）。
- 模板可见性（public/private）、作者、使用次数统计。

### 3.3 数据模型
```prisma
model Template {
  id            Int      @id @default(autoincrement())
  title         String
  category      String?
  businessType  String?
  coverAssetId  Int?
  canvasJson    Json     // 含占位符标记 {{clientName}} 等
  prompt        String?
  configJson    Json?
  isPublic      Boolean  @default(false)
  authorId      Int?
  useCount       Int      @default(0)
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
  @@index([businessType, isPublic])
  @@index([category])
}
```

### 3.4 接口契约
- `POST /api/projects/:id/versions/:vid/save-as-template` body `{title,category,businessType,isPublic}` → 冻结积分（按 credit-rule `template_save`）→ 写 Template
- `GET /api/templates?businessType=&category=&q=&page=` → 公开模板列表
- `POST /api/templates/:id/apply` → 新建 Project + 首个 ProjectVersion（载入 canvasJson），useCount++
- `GET /api/templates/:id` / `PATCH /api/admin/templates/:id`（复用后台模板管理）

### 3.5 验收标准
- 从项目版本「保存为模板」后，新模板出现在列表；「套用」生成可编辑新项目。
- 使用次数随套用递增；积分正确冻结/消费/退款。

### 3.6 测试场景
- **单元**：占位符提取 `extractPlaceholders(canvasJson)`。
- **接口**：保存为模板走积分冻结（mock worker 成功→消费，失败→退款，断言 DB 真值）。
- **安全**：套用他人 private 模板→403。
- **E2E**：保存→列表→套用→编辑→导出，全流程真数据。

## 4. M6 生图 API 统一适配层 + 局部重绘/扩图【P0】

### 4.1 目标
将 gpt-image-2 / 通义万相 / 可灵 / FLUX 抽象为**同一异步任务接口**（提交-轮询-Webhook），并新增**局部重绘（inpainting）、扩图（outpainting）、智能抠图**作为生图后的编辑算子。

### 4.2 功能点
- **Provider Adapter 抽象**：`ImageProvider` 接口 `{ submit(req): jobId, getStatus(jobId), webhook(event) }`；实现 `GptImage2Provider`（现有网关）、`WanxiangProvider`、`KelingProvider`、`FluxProvider`（按 `.env` 启用）。
- 统一任务表 `GenerationJob` 增加 `provider`、`model`、`parentJobId`、`editType`（generate|inpaint|outpaint|remove-bg）。
- 局部重绘：前端在画布框选区域 + 提示词 → `POST /api/image-jobs` body `{editType:'inpaint', maskImage, prompt, baseAssetId}` → Worker 调 provider inpaint → 回写资产。
- 扩图：body `{editType:'outpaint', assetId, direction, prompt}`。
- 智能抠图：body `{editType:'remove-bg', assetId}` → 调抠图 provider → 透明 PNG。

### 4.3 数据模型
```prisma
// 在 GenerationJob 增加：
// provider   String  @default("gpt-image-2")
// model      String?
// parentJobId Int?
// editType   String  @default("generate") // generate|inpaint|outpaint|remove-bg
// maskAssetId Int?
```
新增 provider 配置表（密钥存 `.env` 或密钥服务，库只存 enabled/priority/base_url/model）：
```prisma
model ImageProviderConfig {
  id        Int      @id @default(autoincrement())
  provider  String   @unique
  displayName String
  baseUrl   String?
  model     String?
  enabled   Boolean  @default(false)
  priority  Int      @default(0)
  updatedAt DateTime @updatedAt
}
```

### 4.4 接口契约
- `POST /api/image-jobs` 扩展 body：
```json
{ "projectId": 1, "editType": "inpaint",
  "prompt": "...", "baseAssetId": 10, "maskAssetId": 11,
  "provider": "auto" }
```
- `GET /api/image-jobs/:id`（不变，返回统一状态）
- `GET /api/admin/image-providers` / `PATCH /api/admin/image-providers/:provider` → provider 启停/优先级（admin）
- Worker 依据 `editType` 路由到 `provider.inpaint/outpaint/removeBg/generate`。

### 4.5 关键流程
提交→冻结积分（按 editType 对应 credit-rule）→ 入 `image-generation` 队列 → Worker 选 provider（priority，auto=取最高 priority 且 enabled）→ 调外部 → 成功回写 Asset + consume 积分；失败 refund + 死信队列。

### 4.6 验收标准
- 切到万相/可灵/FLUX provider（`.env` 配置）能跑通生图；`provider` 字段正确记录。
- inpaint/outpaint/remove-bg 三类编辑算子产出正确资产并扣分。
- provider 全部不可用时，前端 mock 分支兜底（复用现有 runMode==='mock'）。

### 4.7 测试场景
- **单元**：provider 选择 `selectProvider('auto', configs)`；editType→积分映射。
- **集成**：Worker 用 mock provider 跑 generate/inpaint/outpaint/remove-bg 四种，断言资产写入 + 积分真值。
- **安全**：越权提交他人 projectId → 403；积分不足 → 400（InsufficientBalanceError）。
- **性能**：provider 超时 → 任务 failed + 退款，不卡死。

### 4.8 上线检查
- 迁移加字段；`credit-rules` 增加 inpaint/outpaint/remove-bg 单价（默认与 generate 同）。
- Worker 编译产物 `worker:prod` 重启；provider 配置从 `.env.production.example` 读取，密钥勿入库。

## 5. M7 设计工作台增强（实时协同 + MCP 扩展 + UI 性能）【P0】

### 5.1 目标
在现有 Fabric 双画布（生产+创意）基础上，补：实时协同（光标/选区/在线状态）、**插件/MCP 窄接口架构（cowart 经验）**、组件/Design Token、**评论批注**、并修复当前 4 个 UI 问题（滚动条统一、缩略图损坏、AI 需求分析显示）。

### 5.2 功能点
- **实时协同**：基于 Redis Pub/Sub（房间 `project:{id}`）广播光标/选区/在线成员；首期不做像素级合并，仅光标+选区+在线状态（避免与 Fabric 串行化冲突）。前端用 WebSocket（现有进度推送通道复用）接收。
- **MCP 窄接口（cowart 经验）**：定义少量 MCP 工具：`insert_generated_image(projectId, assetId, x, y)`、`get_selection(projectId)`、`apply_edit_from_annotation(projectId, annotation)`；选区每 250ms 写 `cowart-selection.json` 到项目缓存（Redis），让 agent 读结构化 JSON 而非截图。
- **组件/Design Token**：在 BrandKit（M4）之上，前端增加 `design-tokens.ts`（颜色/字体/圆角/间距/动效时长家族），全局替换硬编码样式，**统一滚动条**（`::-webkit-scrollbar` + Firefox `scrollbar-*` 自定义，适配系统主题，禁用原生浏览器弹框）。
- **评论批注**：画布内评论（x,y,text,author,resolved），`@` 人通知。
- **性能修复**：缩略图改用 Web Worker（或后端 sharp 预生成）避免主线程阻塞与损坏；AI 需求分析（brief）结果在面板正确渲染（当前显示异常）。

### 5.3 数据模型
```prisma
model CanvasComment {
  id        Int      @id @default(autoincrement())
  projectId Int
  project   Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)
  authorId  Int
  x         Float
  y         Float
  text      String
  resolved  Boolean  @default(false)
  createdAt DateTime @default(now())
  @@index([projectId, resolved])
}
// 协同在线状态/光标存 Redis（不落库），key: collab:project:{id}:peers
```

### 5.4 接口契约
- `WS /api/projects/:id/collab` → 订阅光标/选区/在线事件（复用现有 WS 鉴权）
- `POST /api/projects/:id/comments` / `GET /api/projects/:id/comments` / `PATCH /api/comments/:id/resolve`
- MCP 工具由服务端 `apps/server/src/mcp/` 暴露（stdio/http），供 agent 调用；`get_selection` 读 Redis `collab:project:{id}:selection`

### 5.5 验收标准
- 两个浏览器标签打开同一项目，能看到对方光标与在线状态（不要求实时图层合并）。
- 评论可新增/定位/解决；统一滚动条在所有页面生效，无原生弹框。
- 缩略图在所有列表正确渲染；AI 需求分析面板正常显示。

### 5.6 测试场景
- **单元**：`design-tokens` 一致性校验（无硬编码色值泄漏）；`cowart-selection.json` 序列化/反序列化。
- **集成**：WS 协同频道鉴权（非法 token 拒连）；评论 CRUD 归属校验。
- **E2E**：双标签协同光标可见；评论全流程；滚动条视觉走查（截图比对）。
- **兼容**：Chrome/Edge/Firefox 滚动条样式一致。

### 5.7 上线检查
- Redis Pub/Sub 频道命名空间隔离；WS 鉴权复用 `auth` 中间件。
- 前端 `design-tokens` 全量替换；原生 `alert/confirm` 改为自定义弹框组件。

## 6. M10 合规与审核（NSFW + 广告法黑词）【P0】

### 6.1 目标
在生成/上传链路前置**机审**：NSFW/敏感内容审核 + 广告法黑词/行业资质拦截，降低拒稿与合规风险。

### 6.2 功能点
- **NSFW 审核**：生图/上传资产后，调审核服务（可接 fal `safety_checker` / 自建 / 第三方）。命中则标记 `asset.reviewStatus='rejected'` 并理由。
- **广告法黑词**：维护 `BlackWord` 库（绝对化用语「最佳/第一」、医疗违禁词等），对 AI brief 文本、画布文字（OCR 旁路已具备）、项目备注做扫描，命中高亮+拦截。
- 审核前置到生成链路：brief 含黑词→提示修改；生图结果 NSFW→不入库/标脏。
- 审核报告：资产/项目携带 `reviewStatus`（pending/passed/rejected）+ `reviewReason`。

### 6.3 数据模型
```prisma
model BlackWord {
  id        Int      @id @default(autoincrement())
  word      String   @unique
  category  String   // absolute | medical | industry | custom
  severity  String   @default("block") // block | warn
  createdAt DateTime @default(now())
}
// Asset 增加：reviewStatus String @default("pending"); reviewReason String?
// Project 增加：reviewStatus String @default("pending")
```

### 6.4 接口契约
- `GET/POST/DELETE /api/admin/black-words`（admin）
- `POST /api/ai/brief` 返回增加 `flaggedWords[]`（warn 提示，block 拦截返回 400）
- `GET /api/assets/:id/review` → 审核状态；worker 在写入资产后自动触发审核任务
- `POST /api/admin/review/:assetId` → 人工复核覆盖

### 6.5 验收标准
- 生图结果命中 NSFW → 资产 `reviewStatus=rejected`，前端不可导出/交付。
- brief 含「最佳」等黑词 → 返回命中词并拦截（block）或提示（warn）。
- 管理员可维护黑词库并人工覆盖误杀。

### 6.6 测试场景
- **单元**：`scanBlackWords(text, words)` 返回命中与等级。
- **集成**：上传资产→审核 worker→NSFW 命中→资产标脏且不可导出（断言 DB）。
- **安全**：被拒资产导出接口返回 403。
- **E2E**：正常图通过；违规图拦截并提示。

### 6.7 上线检查
- 迁移加 `reviewStatus` 字段与默认值 `pending`；`BlackWord` 种子（常见广告法违禁词）。
- 审核服务不可用（第三方挂）→ 降级为 `pending` 人工复核，不阻断主链路。

## 7. M13 订单与履约（核价 + 印前预飞 + 工单）【P0】

### 7.1 目标
把当前本地 `orders.json`（Node 写入 + sanitize 防损）升级为**数据库订单 + 在线核价 + 印前预飞 + 生产工单状态机**，形成实物交付闭环。

### 7.2 功能点
- **在线核价引擎**：规则 `PricingRule`（材质 × 尺寸 × 工艺 → 单价公式），输入材质/长宽/数量/工艺 → 输出报价；当前端招牌场景用「公分×单价」公式。
- **印前预飞**：导出文件自动校验（出血≥3mm、分辨率≥300DPI、色彩模式、字体嵌入/转曲、总墨量 TAC≤300%），输出预检报告；不满足则提示。
- **生产工单状态机**：`Order.status` = `draft→quoted→confirmed→production→qc→shipped→signed→done`（可回退/取消），进度对客户可见。
- **材质工艺库**：`MaterialProcess`（喷绘布/亚克力/UV/发光字/雪弗板 + 工艺参数与图例）。
- **客户审阅确认**：confirm 前设计稿留痕（关联 ProjectVersion）；确认后锁版。
- **物流 + 开票对账**（P2）：送货单、电子回签、对账单（本期可先留接口）。

### 7.3 数据模型
```prisma
model Order {
  id            Int      @id @default(autoincrement())
  projectId     Int
  project       Project  @relation(fields: [projectId], references: [id])
  ownerId       Int
  status        String   @default("draft") // draft|quoted|confirmed|production|qc|shipped|signed|done|canceled
  quantity      Int      @default(1)
  material      String   // 引用 MaterialProcess.code
  processJson   Json?    // 工艺参数
  widthCm       Float
  heightCm      Float
  priceCents    Int?     // 核价结果（分）
  currency      String   @default("CNY")
  versionId     Int?     // 客户确认锁定的版本
  confirmedAt   DateTime?
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
  @@index([ownerId, status])
}

model PricingRule {
  id        Int      @id @default(autoincrement())
  material  String
  process   String?
  basePriceCents Int
  perUnitCmCents  Float  // 每平方公分单价
  minPriceCents  Int      @default(0)
  enabled   Boolean  @default(true)
  @@unique([material, process])
}

model MaterialProcess {
  id        Int      @id @default(autoincrement())
  code      String   @unique
  name      String
  category  String   // 喷绘|亚克力|UV|发光字|雪弗板
  specJson  Json?
  exampleAssetId Int?
}

model PreflightReport {
  id        Int      @id @default(autoincrement())
  orderId   Int      @unique
  passed    Boolean
  issues    Json?    // [{check:'bleed',ok:false,detail:'出血不足3mm'}]
  createdAt DateTime @default(now())
}
```

### 7.4 接口契约
- `POST /api/orders/quote` body `{material,process,widthCm,heightCm,quantity}` → `{priceCents, currency, breakdown}`
- `POST /api/orders` body `{projectId, material, process, widthCm, heightCm, quantity, versionId?}` → 建单（status=draft→quote 并存价）
- `POST /api/orders/:id/confirm` → 锁版（校验 versionId）+ status=confirmed
- `POST /api/orders/:id/preflight` → 跑预飞（Worker 调 sharp 检测分辨率/尺寸，PDF 解析出血/字体）→ 写 PreflightReport
- `PATCH /api/orders/:id/status` body `{status}` → 工单流转（admin/工人），写状态历史
- `GET /api/orders?status=` / `GET /api/orders/:id`
- `GET /api/materials` / `GET /api/admin/pricing-rules`（admin CRUD）

### 7.5 验收标准
- 输入材质/尺寸/数量 → 报价与 `PricingRule` 公式一致（可单测断言）。
- 导出 PDF 预飞：出血不足/分辨率不足 → `PreflightReport.passed=false` + 具体 issues。
- 工单状态机合法流转；越权改他人订单状态→403；客户确认后版本锁定不可改。

### 7.6 测试场景
- **单元**：`calcPrice(rule, {w,h,qty})` 边界（qty=0、超大尺寸、minPrice 兜底）；`runPreflight(meta)`。
- **集成**：建单→报价→确认→预飞→生产→发货 全状态机（含非法跳变断言 400）。
- **安全**：工人改他人订单→403；客户取消已 shipped 订单→400。
- **E2E**：真实项目建单核价→确认→预飞报告→状态推进。

### 7.7 上线检查
- 迁移含 `Order/PricingRule/MaterialProcess/PreflightReport` + 索引；`PricingRule` 种子（招牌/喷绘/亚克力单价）。
- 迁移 `orders.json` 历史数据（如有）入 `Order`；保留 `sanitize()` 在写入入口防 U+FFFD。
- 工单状态历史表（可选 `OrderStatusHistory`）便于追踪。

## 8. M15 监控运维与告警【P0】

### 8.1 目标
补齐指标采集、日志聚合、告警，保障生产稳定性。

### 8.2 功能点
- **指标**：任务失败率、队列积压（BullMQ `getJobCounts`）、网关可用率（provider 探活）、积分异常波动（大额/高频）、接口错误率/延时（中间件计时）。
- **日志**：结构化日志（已有 morgan）统一 JSON；错误上报（Sentry 或自研接收端）。
- **告警**：阈值触发（如失败率>5%、队列积压>100、网关连不上）→ 渠道（Webhook/邮件/企微机器人）。
- **健康检查**扩展：`/health` 增加 PG/Redis/Worker 存活与队列深度。

### 8.3 实现要点（不新增重依赖，优先 Redis 计数器 + 轻量）
- `middleware/metrics.ts`：每次请求记 `req_counter{route,status}`、`req_duration_ms`（Histogram 用 Redis sorted set 或简单滑动窗口）。
- `workers/health.probe.ts`：定时（`setInterval`）探活 provider + 采集 BullMQ counts → 写 Redis `metrics:*`，过期滚动。
- `services/alert.service.ts`：读阈值配置（`SystemSetting` 或 env），触发时 POST 告警 Webhook。
- `GET /api/admin/metrics`（admin）返回聚合；`/health` 返回 `{pg,redis,worker,queueDepth}`。

### 8.4 验收标准
- 压测下 `/api/admin/metrics` 能反映失败率/队列深度；provider 挂→告警 Webhook 收到消息；`/health` 正确反映依赖健康。

### 8.5 测试场景
- **单元**：阈值判定 `shouldAlert(metric, threshold)`。
- **集成**：制造队列积压（入 200 任务不消费）→ metrics 显示积压；provider 返回 500→探活失败→告警触发。
- **E2E**：`/health` 在 PG 宕机时返回 503 部分降级。

### 8.6 上线检查
- 告警 Webhook URL 走 env（不入库）；指标采集开关 env 控制，避免性能损耗。
- 生产 `worker:prod` 同时负责 probe（或独立 `metrics:probe` 进程）。

## 9. M14 开放平台与 API【P0】

### 9.1 目标
将生图/合成/导出/模板/资产能力以**对外 REST API + Webhook + OAuth2** 暴露，支持嵌入客户 ERP/门店系统。

### 9.2 功能点
- **OAuth2 Client 凭证模式**：`ApiClient`（clientId/secret，scopes）+ 令牌发放 `/api/oauth/token`。
- **开放端点**（鉴权走 Bearer token）：`POST /open/v1/image-jobs`、`GET /open/v1/image-jobs/:id`、`POST /open/v1/composition-jobs`、`POST /open/v1/projects/:id/export`、`GET /open/v1/templates`。
- **Webhook**：任务完成/失败回调客户 `callbackUrl`（签名 HMAC）。
- **用量计量**：按 ApiClient 计调用次数，可配额限流（复用 Redis 限流器分桶）。
- **Embed**：可选 iframe 嵌入最小工作台（后续）。

### 9.3 数据模型
```prisma
model ApiClient {
  id          Int      @id @default(autoincrement())
  name        String
  clientId    String   @unique
  clientSecretHash String
  scopes      String   // JSON 数组
  quotaPerDay Int      @default(1000)
  createdBy   Int
  createdAt   DateTime @default(now())
  @@index([createdBy])
}
model ApiToken {
  id        Int      @id @default(autoincrement())
  clientId  String
  tokenHash String
  expiresAt DateTime
  revokedAt DateTime?
  @@index([clientId])
}
model WebhookEvent {
  id        Int      @id @default(autoincrement())
  clientId  String
  event     String   // job.succeeded | job.failed
  payload   Json
  signature String
  deliveredAt DateTime?
  createdAt DateTime @default(now())
}
```

### 9.4 接口契约
- `POST /api/oauth/token` body `{grant_type:'client_credentials', client_id, client_secret, scope}` → `{access_token, expires_in}`
- `POST /open/v1/image-jobs`（Bearer）→ 同内部，返回 jobId
- `POST /open/v1/export` body `{projectId, format, callbackUrl}` → 任务完成后 POST callbackUrl（HMAC 签名）
- `GET /open/v1/usage` → 当日用量

### 9.5 验收标准
- 第三方用 client_credentials 拿到 token，调用开放端点成功；任务完成收到签名 Webhook。
- 超配额→429；无效 scope→403。

### 9.6 测试场景
- **安全**：无 token→401；越权 scope→403；Webhook 签名校验失败→拒收（mock 接收端）。
- **集成**：开放端点复用内部生图链路（断言资产写入、积分扣减）。
- **限流**：超 quotaPerDay→429。

### 9.7 上线检查
- `clientSecret` 仅创建时返回一次，库存 hash；Webhook 签名密钥 env。
- 开放端点复用现有 `credit.service` 与所有权校验，不另起炉灶。

## 10. M3 订阅计费 + 支付充值【P0】

### 10.1 目标
在虚拟积分之上，补**真实支付充值 + 订阅套餐 + 发票/对账**，实现商业化闭环。

### 10.2 功能点
- **套餐**：`Plan`（free/pro/team，含每月赠送积分、单价折扣、席位）。
- **支付充值**：微信/支付宝/对公，下单→支付网关回调→`recharge` 流水→积分入账（走 `credit.service` 充值通道）。
- **订阅**：按月/年扣费（可接支付分期或到期提醒），套餐影响 `role-config` 折扣。
- **发票/对账**：充值记录可开票；管理员对账报表（充值 vs 消费 vs 退款）。
- **账务对账**：每日 `CreditTransaction` 汇总与支付流水对账，差异告警（接 M15）。

### 10.3 数据模型
```prisma
model Plan {
  id        Int      @id @default(autoincrement())
  code      String   @unique  // free|pro|team
  name      String
  monthlyCredits Int
  priceCents Int
  discount  Float   @default(1) // 积分单价折扣
  seats     Int      @default(1)
}
model Subscription {
  id        Int      @id @default(autoincrement())
  userId    Int
  planCode  String
  status    String   @default("active") // active|canceled|past_due
  currentPeriodEnd DateTime
  createdAt DateTime @default(now())
}
model PaymentOrder {
  id          Int      @id @default(autoincrement())
  userId      Int
  amountCents Int
  channel     String   // wechat|alipay|offline
  status      String   @default("pending") // pending|paid|failed|refunded
  creditsGranted Int
  gatewayNo   String?  // 支付平台单号
  paidAt      DateTime?
  createdAt   DateTime @default(now())
  @@index([userId, status])
}
// CreditTransaction.type 增加 'recharge' | 'subscription_grant'
```

### 10.4 接口契约
- `GET /api/plans` / `POST /api/subscriptions` / `PATCH /api/subscriptions/:id/cancel`
- `POST /api/billing/recharge` body `{amountCents, channel}` → 返回支付二维码/链接
- `POST /api/billing/callback/{channel}` → 支付网关回调（验签）→ 入账 + 流水
- `GET /api/billing/invoices` / `POST /api/admin/billing/reconcile`（admin 对账）

### 10.5 验收标准
- 充值成功→积分按 `recharge` 入账（断言 DB）；支付失败→不入账。
- 订阅到期→套餐降级 free；对账报表充值=消费+退款+余额变动一致。

### 10.6 测试场景
- **单元**：`grantCreditsOnPaid(order)` 幂等（重复回调不重复入账）。
- **集成**：mock 支付回调→积分入账+流水；越权查他人发票→403。
- **安全**：回调验签失败→拒绝；金额篡改→拒绝。
- **E2E**：充值→生图消耗→余额一致。

### 10.7 上线检查
- 支付密钥 env；回调验签；`recharge` 幂等（gatewayNo 唯一）。
- 与 M15 对账告警联动。
