## 4. 多 Agent 分工建议（直接照此派活）

将 03 的每个 P0/P1 模块作为一个独立开发 Agent 任务，共享同一代码库分支（建议 `feat/benchmark-modules`），遵循 03 §0 通用约束：

| Agent | 负责模块 | 关键依赖 |
|---|---|---|
| Agent-权限 | M2 | 无（先动） |
| Agent-前端修复 | M7(UI部分) | 无（先动，纯前端） |
| Agent-品牌 | M4 | 无 |
| Agent-模板 | M5 | M4（brandKitId 可选） |
| Agent-生图 | M6 | 现有 Worker / credit.service |
| Agent-合规 | M10 | M6 资产写入后触发审核 |
| Agent-履约 | M13 | credit.service（报价不涉及积分）、sharp |
| Agent-监控 | M15 | Redis、BullMQ |
| Agent-开放API | M14 | M6/M13 内部链路 |
| Agent-支付 | M3 | credit.service、PaymentOrder 幂等 |
| Agent-批量 | M8(P1) | M5 模板、M6 Worker |
| Agent-协作 | M16(P1) | M7 评论 |

**协调机制**：
- 所有 Agent 复用 `credit.service`、`auth/rbac` 中间件、BullMQ 队列、`SystemSetting`（credit-rules）。
- 数据模型增量在 `schema.prisma` 各自追加，**迁移文件各自生成并 `migrate deploy`**，避免冲突（建议一人统一收口迁移，或每个 Agent 仅加自己表、由负责人 merge 后统一生成迁移）。
- 每 Agent 交付：代码 + 单元测试 + 接口/集成测试 + 真实数据 E2E（复用 `_e2e_final.mjs` 模式）+ 迁移 SQL + 验收清单。

## 5. 验收与上线门槛（统一）

- 后端 `tsc --noEmit` 通过。
- 对照 03 每个模块的「测试场景」全部绿（单测/集成/安全/E2E）。
- 真实数据 E2E：登录→建项目→生图/合成→导出，积分前后端一致（参考 `TEST_REPORT_2026-07-14.md`）。
- 迁移 `migrate deploy` 成功；`prisma/checks.sql` 已执行（余额非负兜底）。
- 生产 `worker:prod` 起独立进程；监控（M15）能看到队列深度与失败率。
- 敏感字段（M2）越权访问测试必含反例。

---

## 6. 一句话结论

本系统是**广告 AI 设计生产交付**赛道里「主业务闭环已跑通、行业标配能力大面积缺失」的状态。补齐 M2/M4/M5/M6/M7/M10/M13/M15/M14/M3 十个 P0 模块后，即达到与头部同类产品可比的基线；其中**品牌套件、模板转模板、生图适配层、合规机审、订单履约（核价/预飞/工单）、开放 API、监控、支付**是「不做就不算同类产品」的必做题。03 文档已细化到可直接派发开发 Agent 上手。
