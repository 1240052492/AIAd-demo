# AdCraft AI（神笔）开发需求文档

更新时间：2026-07-14

## 1. 项目定位

本项目是一个面向广告行业的 AI 设计生产与交付系统，目标是让用户从客户需求出发，完成广告 Brief、AI 生图、环境合成、画布编辑、重新生成、积分消耗、后台管理与交付导出。

当前项目已经从“方案/原型阶段”进入“可验收 MVP + 生产加固后待优化”阶段。主业务链路已经跑通，后续重点是上线配置、商业化能力、稳定性监控和规模化扩展。

## 2. 当前已完成内容

### 2.1 核心业务闭环

## 4. 当前存在的问题

### 4.1 文档状态与代码状态不完全一致

`docs/superpowers/plans/2026-07-13-infinte-image-workbench-integration.md` 中的任务 checkbox 仍显示未完成，但实际抽查关键文件已经存在。后续需要统一更新计划文档，否则新开发者容易误判当前进度。

### 4.2 README 和报告在 PowerShell 下存在中文乱码显示

部分 Markdown 文件在 PowerShell `Get-Content` 输出时出现乱码，推测是编码显示问题。文件本身可能是 UTF-8，但建议统一检查并确保仓库文档全部使用 UTF-8，无 BOM 或统一 BOM 策略。

### 4.3 高并发性能仍需优化

根据 `PERFORMANCE_REPORT_2026-07-13.md`：

- 正常 50 VU 负载下多数接口 p95 小于 100ms。
- `assets` 端点 p95 约 202ms，需优化。
- 200 VU 压力下所有端点 p95 超过 400ms，尚不满足高并发 SLA。
- 当前瓶颈主要来自单节点数据库容量和测试共置环境。

### 4.4 生产上线依赖真实环境配置

上线前仍需要确认：

- 真实 `DATABASE_URL`、`REDIS_URL`、JWT 密钥、网关密钥。
- Docker Compose 或生产容器编排配置。
- 是否启用 OCR sidecar。
- 文件存储是否继续本地落盘，还是切到 MinIO/对象存储。
- 支付和发票等商业化流程是否纳入本期。

## 5. 需要优化的部分

### 5.1 产品体验优化

- 梳理首页工作台从“需求输入 → 策略 → 生图 → 合成 → OCR 校验 → 修正 → 导出”的用户路径。
- 明确每一步的失败提示、重试入口和积分预估。
- 管理端应提供更清晰的任务失败原因、队列状态、网关状态和积分异常提示。
- 编辑器可继续增强模板套用、图层管理、导出格式和历史版本体验。

### 5.2 工程质量优化

- 更新工作台集成计划文档 checkbox，与真实代码状态对齐。
- 增加工作台相关 API 的集成测试。
- 增加 OCR、文字修正、SVG 安全过滤的回归测试。
- 将性能测试脚本和阈值纳入 CI 或发布检查。
- 清理临时审计产物和截图产物，明确哪些文件应纳入版本库。

### 5.3 安全与合规优化

- 接入真实支付前，需要完成账务对账、退款、异常补偿和管理员操作审计。
- SVG 上传/保存需要持续加强安全过滤。
- 上传图片需要结合业务要求决定是否启用 NSFW/敏感内容审核。
- 管理后台需要细化角色权限和操作日志。

### 5.4 部署与运维优化

- 接入对象存储和 CDN，避免生产环境依赖本地磁盘。
- 增加监控告警与日志采集。
- 增加数据库连接池和扩展方案。
- 验证 Redis 不可用、网关不可用、OCR 不可用、Worker 崩溃等异常场景。

## 6. 建议下一步

建议按以下顺序推进：

1. 同步更新工作台集成计划文档，把已完成项勾选，明确未完成项。
2. 跑一次当前代码的完整构建和端到端验证，形成新的测试报告。
3. 决定上线范围：仅 MVP 试用，还是包含真实支付和对象存储。
4. 若准备上线，优先完成生产环境变量、Docker 部署、对象存储、监控告警。
5. 若继续开发功能，优先补工作台 OCR/文字修正/SVG 导出的测试和体验细节。

## 7. 给后续开发者的注意事项

- 不要重复实现已经存在的积分、任务、资产和 Worker 链路。
- 新增任何消耗积分的功能，都必须走统一积分规则、冻结、消费、退款和流水记录。
- 涉及用户资产、项目、任务、导出的接口，必须做所有权校验，避免越权访问。
- 修改首页工作台时，需要同时检查前端 API 客户端、服务端路由、Worker 和数据库模型。
- 不要把真实密钥、token、数据库连接串提交到仓库。
- 多 PC 开发时，优先同步 Git 远端和本文件，再继续开发。
